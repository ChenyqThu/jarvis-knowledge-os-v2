## 1 · Context — what's already true

Filesystem-canonical **Steps 1-2.3 are done**. The write path, dream
cron, slug normalization, Step 1.6 round-trip audit, and KOS→gbrain
frontmatter mapping all landed between 2026-04-22 and 2026-04-24
(see `skills/kos-jarvis/TODO.md` § Filesystem-canonical entry +
`docs/JARVIS-ARCHITECTURE.md §§ 6.11 / 6.13`).

**What didn't land**: the bulk export of the 1857 pages that live
only in PGLite. Today:

- `~/brain/` has **110 `.md`** files (notion sources from the post
  Step-2.2 HTTP-ingest path + a handful of orphan-reducer sentinel
  writes).
- PGLite has **1967 pages**. The delta of **1857 pages** is the
  v1-wiki import plus auto-generated stubs from enrich-sweep and
  idea-ingest.

Consequence shape:

| Surface | Impact of the gap |
|---|---|
| `orphan-reducer --apply` | 95% of candidate markdown writes fall through (`markdown_reason: "no_file"`). Only the DB edge lands. |
| `gbrain dream` → `backlinks` phase | Can't walk markdown bodies → can't synthesize reverse-link frontmatter for 95% of the brain. |
| `enrich-sweep` | Tier-1 frontmatter edits happen on disk; 95% of pages don't exist there to be edited. |
| Karpathy LLM-wiki | `grep` / `find` / `git log` work on 110 pages, miss the bulk. |
| Git history | Knowledge VCS only tracks what's been ingested post-Step-2.2 + today's backfills. |

Everything downstream of P1-A is quietly degraded. This is why it's
worth one focused session rather than spliced in.

**What it's NOT**: a re-architecting. The machinery is already there.
`gbrain export --dir <path>` exports all 1967 pages in gbrain 20-dir
MECE layout with KOS frontmatter preserved (verified 2026-04-24 dry-
run to `/tmp/export-probe`). The work is: run export + reconcile
against the 110 files already on disk + commit cleanly to git.

## 2 · Known gotchas (surfaced during 2026-04-24 preflight)

These are the landmines the next session needs to know about.

### 2.1 Duplicate-dir bug in export output

The dry-run export produced:

```
/tmp/export-probe/sources/notion/...        ← expected
/tmp/export-probe/sources/sources/notion/...  ← unexpected, double-prefix
```

Cause: some PGLite rows have `slug = "sources/notion/<id>"` (correct
shape after Step 1.5) but others have `slug = "sources/sources/notion/<id>"`
(carry-over from earlier importer that didn't strip the `sources/`
source_id prefix). Step 1.5 fixed root-level strays; this is a deeper
slug shape bug.

**Decide first** whether to:
- (a) Normalize the bad slugs in-place (PGLite UPDATE) before export —
  cleanest, but touches DB.
- (b) Post-process the export output to dedupe `sources/sources/` →
  `sources/` — simpler, doesn't touch DB.
- (c) Walk the 110 existing disk files first, build a "known good"
  set, and when the export has a duplicate path, prefer disk.

Recommended: **(a)**. Add a Step 1.7 in the plan — a DB-side cleanup
query that rewrites double-prefixed slugs. Mirrors the Step 1.5
pattern.

### 2.2 Frontmatter `related:` uses relative `.md` paths

Sample exported frontmatter:

```yaml
related:
  - ../concepts/context-rot.md
  - ../concepts/context-engineering.md
  - ../syntheses/harness-engineering-landscape.md
```

These are generated by the gbrain export path. They assume the 20-dir
MECE layout. If we place exports into `~/brain/` on the same layout,
they resolve. But they'll be markdown links in raw frontmatter — worth
confirming `gbrain sync` + upstream `extract-links` handle these
correctly (or whether they should be rewritten to `[[slug]]` wikilinks
before landing).

**Action**: grep 10 random exported files for `related:` shape, diff
against an existing disk file that landed via `/ingest` (which uses
the kos-compat-api frontmatter builder with `source_refs:` instead).
Mismatch = design decision point.

### 2.3 Existing disk files may have drifted from DB

The 110 disk files were either:
- Ingested via `/ingest` (frontmatter built by kos-compat-api, has
  `type:` + `title:` after today's P1-D fix)
- Written via `orphan-reducer` (adds sentinel block `<!-- orphan-reducer-inbound -->`
  inside the body)
- Hand-edited? (unlikely but possible)

When `gbrain export` emits a file whose path matches an existing disk
file, the CONTENT may differ because:
- DB has `compiled_truth` + `timeline`, export joins them into body.
  Disk has the original `/ingest` body format with its `> Ingested via`
  comment block.
- Disk pages have orphan-reducer sentinel blocks that DB doesn't
  know about.

**Action**: in Phase 2 (dry merge), explicitly categorize file state:
NEW / MATCH-identical / DIFF-drifted. DIFF cases need per-file review.
My guess: zero DIFF (disk files were last-written and the DB reflects
them via sync), but don't assume.

### 2.4 Large commit to `~/brain` git repo

1857 new files × ~5KB avg ≈ 9MB tree add. Git handles it fine but
`git diff` against the commit will be useless. Plan to:
- Add a **clear commit message** with the exact counts + source.
- Optionally split by kind (one commit per domain: people/, companies/,
  concepts/, projects/, sources/) — slower but more reviewable
  history.

### 2.5 PGLite safety during export

`gbrain export` opens its own PGLite handle. With my WAL-durability
patch, that should be safe. But:
- Don't run during a notion-poller cycle (5-min cadence) — could
  race with ingest writes. Easiest: `launchctl bootout com.jarvis.notion-poller`
  for the duration, restore after.
- Pre-export rolling backup: `cp -r ~/.gbrain/brain.pglite
  ~/.gbrain/brain.pglite.pre-step3.0-<ts>`.

## 3 · The plan — six phases, PASS/FAIL gates between each

### Phase 0 · Preflight (~20 min)

1. Pause notion-poller: `launchctl bootout gui/$UID/com.jarvis.notion-poller`.
2. Rolling backup: `cp -r ~/.gbrain/brain.pglite ~/.gbrain/brain.pglite.pre-step3.0-$(date +%s)`.
3. `gbrain stats` — record `pages`, `chunks`, `links`, `orphans` as baseline.
4. `gbrain export --dir /tmp/export-v1` — capture 1967 pages.
5. Sanity: `find /tmp/export-v1 -name '*.md' | wc -l` == page count.

**Gate**: file count matches, no export errors in stderr. If `sources/sources/`
double-prefix appears (it will), proceed to Phase 1. If not, skip to Phase 2.

### Phase 1 · Slug normalization round 2 (conditional, ~30 min)

Only if Phase 0 surfaces the `sources/sources/` double-prefix (or any
other slug shape bug).

1. Identify affected rows:
   ```sql
   SELECT slug FROM pages WHERE slug LIKE 'sources/sources/%' ORDER BY slug;
   ```
2. Write a fork-local `slug-normalize-round-2/run.ts` under
   `skills/kos-jarvis/` (mirror `slug-normalize/` pattern). For each
   affected slug, the DB UPDATE is `slug = REPLACE(slug, 'sources/sources/', 'sources/')`.
3. Rewire intra-brain markdown references (if any disk file references
   the old slug).
4. Re-run `gbrain export --dir /tmp/export-v2` — no double-prefix dirs.

**Gate**: `find /tmp/export-v2/sources/sources -type d | wc -l` == 0.
Page count still 1967.

### Phase 2 · Dry merge analysis (~15 min)

Walk both trees:

```bash
#!/bin/bash
# Produces docs/plans/P1-A-merge-categorization.md with:
# - NEW (in export, not in ~/brain) count + sample paths
# - MATCH (identical frontmatter+body) count
# - DIFF (both sides, differ) count + full list
```

Write a one-shot classification script that for each export file:
- If no `~/brain/<path>` → NEW.
- Else hash both; if equal → MATCH.
- Else → DIFF, capture the diff for review.

**Gate**: DIFF count is reviewable (target: <20). If >50, figure out
what's drifting first. Likely sources: orphan-reducer sentinel blocks
(disk has, DB doesn't) and kos-compat-api `> Ingested via ...` headers
(disk has, DB doesn't).

### Phase 3 · Apply merge (~15 min mechanical + whatever DIFF review needs)

1. `rsync -av --ignore-existing /tmp/export-v1/ ~/brain/` — copies NEW
   only, preserves existing disk files.
2. For DIFF files: per-file decision. Default: **prefer disk**
   (disk has orphan-reducer sentinels + kos-compat-api headers we
   want to keep). Only accept export-side if disk is clearly stale.
3. `gbrain sync --force --repo ~/brain` — reconcile PGLite. Expected:
   `+0 added, ~0 modified, -0 deleted` (DB is already the source of
   truth). Any modifications here mean disk diverged from DB in
   non-trivial ways; stop and investigate.

**Gate**: `gbrain sync` reports no drift. Stats unchanged.

### Phase 4 · Commit (~5 min)

Single commit or per-domain commits (5-6 commits). Prefer per-domain
for `git log` readability:

```bash
for dir in people companies concepts projects decisions sources syntheses protocols comparisons timelines entities; do
  git -C ~/brain add "$dir" 2>/dev/null
  git -C ~/brain commit -m "Step 3.0: bulk export $dir/ from PGLite (part of P1-A)" 2>/dev/null
done
```

Message template for the main commit:

```
Step 3.0: bulk-export 1857 PGLite-only pages → ~/brain/ filesystem-canonical

Brings every DB row onto disk as .md files per gbrain's 20-dir MECE
layout. Unlocks: orphan-reducer markdown-writes for bulk of brain,
gbrain dream backlinks phase full coverage, enrich-sweep file-based
reconciliation, Karpathy LLM-wiki grep/find/git-log inspection.

Before: 110 .md on disk, 1967 in PGLite.
After: 1967 .md on disk, 1967 in PGLite (1:1).

See docs/plans/P1-A-bulk-export-filesystem-canonical.md for rationale
and docs/SESSION-HANDOFF-P1A-FILESYSTEM-CANONICAL.md for execution
notes.
```

### Phase 5 · Verify (~20 min)

Hard gates before declaring done:

1. `gbrain stats` — pages/chunks/embedded/links/tags/timeline unchanged.
2. `gbrain orphans --count` — unchanged (orphans are structural).
3. `gbrain doctor` — brain_score not worse than before; no new FAILs.
4. `gbrain dream --dry-run --json` — all phases report `status: "ok"`
   or `"warn"` (no `"failed"`). `sync` phase reports 0/0/0 (disk ==
   DB after the merge).
5. Random spot-check: pick 10 .md from 10 different kinds, read each
   to confirm frontmatter integrity (id / kind / type / title present,
   well-formed YAML).
6. Restart notion-poller: `launchctl bootstrap gui/$UID ~/Library/LaunchAgents/com.jarvis.notion-poller.plist`.
7. Watch one poll cycle (5 min) — confirm new ingests still land
   cleanly next to the bulk-exported siblings.

**Gate**: all green. If anything regresses, `git revert <Phase 4 sha>`
and `cp -r ~/.gbrain/brain.pglite.pre-step3.0-<ts> ~/.gbrain/brain.pglite`.

### Phase 6 · Downstream opportunities (incremental, NOT part of P1-A)

After P1-A lands, these become easy:

- **orphan-reducer**: re-run the skill; `skipped_no_file` count should
  drop from 95% to near-zero. Markdown sentinel blocks will start
  appearing across all domains. Good audit-friendly paper trail of
  inbound edges.
- **kos-patrol phase 4**: read file bodies for better entity extraction
  (currently walks DB `compiled_truth`; filesystem has richer context).
- **enrich-sweep**: switch from `gbrain put` shell-out to direct
  markdown edit + `gbrain sync` — aligns with the filesystem-canonical
  write path.

Track these as separate P2 entries post-P1-A.

## 6 · Escape hatches

If Phase 2 shows >100 DIFF files: **stop**. The DB and disk have
diverged in ways worth understanding first. Possible causes:
- Notion-poller ingest races during the session (poller wasn't
  booted out early enough).
- orphan-reducer wrote sentinel blocks faster than the DB reflected
  them (shouldn't happen with WAL-durability fix, but verify).
- Manual edits by a prior session.

If Phase 5 verification shows a regression: **rollback, don't
forward-fix**. The pre-step3.0 PGLite backup + git revert restore
the invariant. Diagnose from the post-mortem snapshot separately.

If the session runs out of time mid-phase: Phase boundaries are
idempotent. Save state in this plan's DONE markers and pick up in
the next session.
